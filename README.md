# Grid-бот для Binance и Bybit (ccxt)

Детерминированный grid-трейдинг-бот: расставляет лимитные ордера на покупку
ниже текущей цены и на продажу выше, зарабатывая на колебаниях цены внутри
заданного диапазона. Поддерживает spot и futures на **Binance** и **Bybit**
через единый интерфейс `ccxt`.

> Здесь нет «AI-стратегий» и подключений к LLM. Это намеренно: алгоритм
> детерминированный, и каждую строчку можно прочитать и понять, что именно
> она сделает с деньгами.

## Как это работает

Пользователь задаёт ценовой диапазон (`LOWER_PRICE`/`UPPER_PRICE`) и число
уровней сетки (`NUM_GRIDS`). Бот:

1. Строит равномерные ценовые уровни внутри диапазона.
2. Ставит лимитные **buy** ниже текущей цены и **sell** выше.
3. Когда исполняется **buy** — ставит **sell** на уровень выше.
4. Когда исполняется **sell** — ставит **buy** на уровень ниже.

Прибыль набегает на колебаниях внутри диапазона. **Если цена устойчиво
выходит за пределы диапазона без ручной корректировки — бот теряет деньги.**

## Создание API-ключей (важно про права)

При создании ключа на бирже включайте **только право на торговлю (trade)** и
**обязательно отключайте право на вывод средств (withdrawal)**. Бот не делает и
не должен делать выводов средств.

- **Binance**: Account → API Management → Create API. Включите *Enable
  Spot & Margin Trading* и/или *Enable Futures*. **НЕ включайте** *Enable
  Withdrawals*. По возможности задайте whitelisting IP.
- **Bybit**: Account → API → Create New Key. Тип *API Transaction*, права —
  только *Trade* (Spot и/или Derivatives). **Без** *Withdraw*.

Для тестнета создавайте отдельные ключи на тестовых площадках:
- Binance Testnet: https://testnet.binance.vision (spot),
  https://testnet.binancefuture.com (futures).
- Bybit Testnet: https://testnet.bybit.com.

Ключи кладутся в `.env` (`API_KEY`, `API_SECRET`) и читаются только из
окружения. В коде они не хардкодятся и не пишутся в логи открытым текстом.

## Порядок запуска (строго рекомендуется именно так)

Идите по шагам и не перепрыгивайте — это защищает ваши деньги.

### Шаг 1. DRY_RUN (по умолчанию, без реальных ордеров)

```env
DRY_RUN=true
USE_TESTNET=true
```

```bash
python -m grid_bot.main
```

Реальные ордера **не** отправляются. Заполнения симулируются локально на основе
**реальной текущей цены** с биржи. Посмотрите логи, убедитесь, что сетка и
поведение вас устраивают.

### Шаг 2. Testnet (реальные, но фейковые ордера)

Создайте тестнет-ключи, пропишите их в `.env`, затем:

```env
DRY_RUN=false
USE_TESTNET=true
```

Бот отправляет настоящие ордера в sandbox-режиме ccxt — деньги фейковые,
механика настоящая.

### Шаг 3. Live с небольшим капиталом

Только после успешных шагов 1–2:

```env
DRY_RUN=false
USE_TESTNET=false
TOTAL_CAPITAL=<небольшая сумма>
```

Начинайте с маленького `TOTAL_CAPITAL`. Бот никогда не использует больше, чем
указано в `TOTAL_CAPITAL`, даже если на балансе биржи больше.

## Защитные механизмы

- **DRY_RUN** (по умолчанию `true`): никаких реальных ордеров, симуляция по
  реальной цене.
- **Kill switch по просадке** (`MAX_DRAWDOWN_PCT`): при падении equity на
  заданный % от стартового бот отменяет все ордера и останавливается.
- **Ограничение капитала** (`TOTAL_CAPITAL`): жёсткий потолок используемых
  средств.
- **Предупреждение о риске ликвидации** (futures): при старте бот грубо
  оценивает, не попадает ли диапазон сетки в зону возможной ликвидации при
  выбранном плече, и громко предупреждает. Это **приближённая оценка**, а не
  точный расчёт биржи.
- **Логирование**: все ордера, заполнения, ошибки и срабатывание kill switch
  пишутся и в консоль, и в файл `grid_bot.log` с таймстампами.

## Мультитикерный режим (портфель тикеров)

При `MULTI_SYMBOL_MODE=true` бот не торгует один заданный `SYMBOL`, а сам
отбирает **портфель** тикеров с биржи и торгует их одновременно, каждый —
своей независимой grid-сеткой (`SymbolGridInstance`).

### Как отбираются тикеры

`screener.py` делает **один** запрос `fetch_tickers()` (важно для лимитов
запросов), фильтрует и ранжирует по трём категориям:

1. **`volume`** — топ-`NUM_TOP_VOLUME` по 24ч-объёму (`quoteVolume`).
2. **`gainer`** — топ-`NUM_TOP_GAINERS` по % роста за 24ч (`percentage`).
3. **`volatile`** — топ-`NUM_TOP_VOLATILE` по амплитуде `(high-low)/last`.

Перед ранжированием отбрасываются: пары не с `QUOTE_CURRENCY`, стейбл-к-стейблу
(`STABLECOINS`), плечевые токены (`LEVERAGED_TOKEN_PATTERNS`), неликвид
(< `MIN_24H_VOLUME_USDT`) и чёрный список (`EXCLUDE_SYMBOLS`). При пересечении
категорий действует приоритет **volume > gainer > volatile**, итог
ограничивается `MAX_CONCURRENT_SYMBOLS`.

### ⚠️ Почему капитал распределяется неравномерно (важно)

Grid-стратегия зарабатывает на колебаниях **внутри диапазона** и теряет деньги
при устойчивом одностороннем движении. Категории **«рост» и «волатильность»** —
это почти по определению тикеры в сильном тренде или с резкими разворотами,
то есть **наименее подходящие** условия для grid-логики. Поэтому:

- тикерам категории **`volume`** даётся **больший** вес капитала
  (`CAPITAL_WEIGHT_VOLUME`, обычно ликвиднее и спокойнее);
- тикерам **`gainer`/`volatile`** — **меньший** вес
  (`CAPITAL_WEIGHT_GAINER`/`CAPITAL_WEIGHT_VOLATILE`) **и более жёсткий
  per-symbol kill switch** (внутренние коэффициенты `0.7`/`0.6` к
  `MAX_DRAWDOWN_PCT`).

Капитал на символ = `TOTAL_CAPITAL × (вес_категории / сумма весов всех активных
символов)`. **Инвариант безопасности:** суммарный капитал по всем символам
**никогда** не превышает `TOTAL_CAPITAL` (проверяется `assert`-ом в коде и
отдельным тестом).

### Ребалансировка и завершение

- Каждые `RESCAN_INTERVAL_HOURS` портфель пересобирается. Новый символ получает
  капитал из **свободного остатка** (что гарантирует инвариант), для него
  считается авто-диапазон через `GRID_RANGE_PCT`.
- Символ, **выпавший** из выборки, не закрывается резко: по `WIND_DOWN_POLICY`
  он либо `hold` (новые уровни не открываются, текущие ордера доигрывают),
  либо `cancel` (ордера сразу отменяются).

### Двухуровневый kill switch

- **Per-symbol** (`MAX_DRAWDOWN_PCT`, с ужесточением для трендовых категорий):
  останавливает **только** проблемный символ.
- **Портфельный** (`MAX_PORTFOLIO_DRAWDOWN_PCT`): если просадка по сумме equity
  всех символов достигает порога — останавливаются **все** символы и отменяются
  все ордера.

> Все защитные механизмы предыдущих этапов (DRY_RUN, лимит капитала,
> предупреждение о ликвидации для futures) продолжают работать на уровне
> каждого отдельного символа.

## Spot vs Futures и единый формат символа

Различия Binance/Bybit и spot/futures (формат символа, `defaultType`/
`category`) инкапсулированы в `exchange.py`:

- `build_exchange()` — задаёт `defaultType` (`spot`/`swap`), для Bybit futures — `defaultSubType=linear`, и **таймаут запросов 15 секунд** (чтобы сбои testnet не блокировали цикл на минуты).
- `resolve_symbol()` — единая точка преобразования символа: для futures
  (linear perpetual) ccxt использует формат `BASE/QUOTE:QUOTE`
  (например `BTC/USDT:USDT`).
- `setup_futures_market()` — при старте в futures-режиме выставляет плечо
  (`set_leverage`) и режим маржи (`set_margin_mode`) с обработкой ошибок.

Жёсткая валидация при старте: если `MARKET_TYPE=spot`, а `LEVERAGE != 1`,
бот не запустится (ошибка конфигурации, а не тихий игнор).

## Риски futures и плеча (читать обязательно)

- **Плечо умножает и прибыль, и убыток.** При плече 10x движение цены на 10%
  против позиции может стереть всю маржу.
- **Ликвидация.** В futures позиция может быть принудительно закрыта биржей с
  потерей маржи. Особенно опасно, когда диапазон сетки шире, чем дистанция до
  ликвидации: цена может ликвидировать позицию, не выходя из диапазона.
  Предупреждение бота — **грубая оценка**, не полагайтесь на него как на точный
  расчёт. Реальная цена ликвидации зависит от поддерживающей маржи, комиссий,
  funding и размера позиции.
- **`isolated` vs `cross`.** `isolated` ограничивает риск маржой конкретной
  позиции; `cross` рискует всем балансом счёта. Для старта безопаснее
  `isolated`.
- **Funding.** У бессрочных контрактов есть funding-платежи, которые могут
  накапливаться и влиять на результат.
- **Начинайте с минимального плеча (1–2x)** и небольшого капитала.

## Веб-дашборд

К боту прилагается локальный веб-дашборд со статистикой и детальной историей
сделок (включая прибыль по каждой паре buy/sell).

### Как это устроено

Бот (`grid_bot/main.py`) и дашборд (`web/app.py`) — **два отдельных процесса**,
которые общаются через одну и ту же SQLite-базу `bot_data.db`:

- бот **пишет** в базу исполненные сделки, снимки equity, важные события и **heartbeat процесса**;
- дашборд **читает** базу и отдаёт данные в браузер, обновляя свой heartbeat.

### Запуск

В одном терминале — бот:

```bash
source .venv/bin/activate
python -m grid_bot.main
```

В другом терминале — дашборд:

```bash
source .venv/bin/activate
uvicorn web.app:app --reload --port 8000
```

Откройте в браузере: **http://127.0.0.1:8000**

Альтернативно дашборд можно запустить напрямую: `python -m web.app`.

### Что показывает

- **Карточки**: текущая equity **по всему портфелю** (сумма по символам),
  общий P&L (абсолютный и %), win rate, всего сделок, текущая просадка от пика,
  статус бота (running/stopped/killed), режим (spot/futures + `SINGLE`/`MULTI`,
  число активных символов, плечо) и breakdown по категориям.
- **Активные тикеры** (мультирежим): таблица символ / категория / стартовая и
  текущая equity / P&L / число сделок / статус (`active` или `wind-down`),
  обновляется при ресканировании. Добавление/удаление символов также попадает
  в ленту событий.
- **График equity curve** (портфельный) с переключением диапазона (24ч / 7д / всё время).
- **Таблица транзакций**: время, сторона, цена, объём, сумма, реализованный
  P&L, статус (demo/live). Сортировка по клику на заголовок, пагинация,
  фильтры по стороне и диапазону дат.
- **Лента событий**: старт/стоп, срабатывание kill switch, ошибки.
- **Процессы**: статус двух процессов — `Grid Bot (main.py)` и `Dashboard (web/app.py)` — с PID, временем старта и последним pulse (heartbeat). Если pulse старше ~45 секунд, процесс помечается как `STALE`.
- Если `DRY_RUN=true` — заметный баннер **«ДЕМО-РЕЖИМ»**.
- Автообновление каждые ~7 секунд (обычный polling, без WebSocket).

### Расчёт прибыли по сделке (FIFO)

Грид-бот не закрывает позицию целиком — прибыль складывается из пар buy/sell на
соседних уровнях. Реализованный P&L считается по алгоритму **FIFO** (`web/pnl.py`):
каждая закрывающая сделка сопоставляется с самой старой ещё не закрытой
противоположной сделкой того же символа, прибыль =
`(цена_продажи − цена_покупки) × объём`. Для futures корректно обрабатывается
шорт (sell открывает короткую позицию, buy её закрывает). Поле `realized_pnl`
равно `null` для открывающей части пары.

### Безопасность дашборда

- Сервер по умолчанию слушает **только `127.0.0.1`** (loopback), не `0.0.0.0` —
  это локальный инструмент для одного пользователя. Не выставляйте его в сеть и
  не пробрасывайте туннели.
- В базе `bot_data.db` **нет** API-ключей/секретов, и backend их никогда не
  возвращает ни в одном ответе.
- Нет аутентификации и удалённого доступа — это сделано намеренно (локальный
  инструмент).

риском полной потери средств. Используете на свой страх и риск. Сначала —
`DRY_RUN`, затем testnet, и только потом реальные деньги в небольшом объёме.
```
